maven基本功能及使用

前言

  1. maven总体说是项目构建工具，maven功能很多，这篇博客只介绍maven在开发中可能会遇到的问题、常用功能和使用。安装过程略过。

maven

  1. setting.xml（maven基本配置文件）
    • 文件读取：默认读取user/.m2/settings.xml，idea中使用内置maven，可做相关属性修改。
    • 常用配置：依赖下载到本地的仓库地址（localRepository）、依赖下载中心（mirrors）、其他可设置代理、私服配置等不常用，不多做说明
  2. pom.xml常用属性：
    • grouuId：一般为“com.公司名”
    • artifactId：一般为项目名
    • version：版本号
    • packaging：打包形式，默认jar
    • properties：自定义变量
    • dependencyManagement：一般存在父Pom里，其子项目用到相同的依赖，依然声明，但不用再做版本声明，以父Pom为准。
    • dependency：声明依赖，引入依赖jar或者内部项目
      • scope：作用域。默认compile，这个属性的实质是在测试到发布的各个阶段过程中对依赖使用的管理，需要对依赖在什么阶段使用有理解。如果不清楚默认即可，默认的负面效果只是打包或编译、执行加入多余依赖。每个作用域的主要区别在于使用的阶段和这个阶段是否把依赖打入成品包或参加编译、执行。
        1. compile：
          • 阶段：所有
          • 效果：编译、执行并打入成品包中。
          • 样例：大部分依赖都是
        2. test：
          • 阶段：测试
          • 效果：只有测试时期加入编译、执行
          • 样例：Junit
        3. runtime：
          • 阶段：除编译之外
          • 效果：和compile对比，除编译之外其他相同
          • 样例：JDBC（不需编译）
        4. provided：
          • 阶段：除打包之外
          • 效果：同上，只是不加入成品包
        5. system：
          • 阶段：同privider
          • 效果：依赖不从maven仓库下载，直接本地定义
        6. 多项目scope的继承关系
          • test和provided不会继承，其他继承
  3. maven生命周期：
    • 每个生命周期中都有多个阶段（phase），每个阶段独立，但phase有先后顺序，而且后面的依赖于前面的phase，依次执行，如下三个生命周期
      • clean：清理上一次生成的文件
      • default：核心，包括编译项目代码，测试代码，打包等一系列重要操作，用于构建项目
      • site：生成相关文档等操作
    • maven允许将插件绑定到生命周期中如常用的打包工具maven-source-plugin，也可以对插件进行相关配置。
